# 엔드필드 공장 플래너 — 프로젝트 스펙 (CLAUDE.md)

> 이 문서는 Claude Code가 프로젝트 폴더를 열 때 자동으로 읽는 컨텍스트 파일입니다.
> 목적: 명일방주: 엔드필드(Arknights: Endfield)의 통합 공업 시스템(AIC) 설비를 자유롭게 배치·연결해볼 수 있는
> 드래그 앤 드롭 기반 "공장 레이아웃 플래너" 웹 도구.
> 조사 기준일: 2026-07-19, 게임 버전 1.4 「심연으로」 기준.
> 설비/자원 명칭은 한국어 정식 서비스 명칭(나무위키 「명일방주: 엔드필드/설비」 문서 대조) 기준.

## 1. 프로젝트 목표

- 사용자가 그리드 위에 공업 설비(채굴기, 정련로, 분쇄기, 부품 가공기, 포장기, 전력, 저장 등)를 자유 배치
- 설비 간 컨베이어 벨트/파이프 연결을 시각적으로 그릴 수 있어야 함
- 전력망(프로토콜 코어 → 중계기 → 전력 공급기) 커버리지를 반경 원으로 시각화
- 배치한 레이아웃 저장/불러오기(JSON export/import, localStorage 자동 저장, 공유 URL)
- 실제 게임처럼 "정확한 좌표/판정"까지 재현할 필요는 없음 — 생산 라인이 병목 없이 맞물리는지
  미리 검증하는 용도의 **기획 도구**

## 2. 구현 현황 (2026-07 기준)

Vite + TypeScript(Vanilla) + HTML5 Canvas로 구현 완료. GitHub Pages 자동 배포(.github/workflows/deploy.yml).

```
index.html                    Vite 진입점
public/data/facilities.json   설비 카탈로그 (런타임 fetch, 코드 수정 없이 값 편집)
src/
  main.ts      부트스트랩          app.ts       UI·캔버스·인터랙션
  types.ts     도메인 타입          catalog.ts   설비 데이터 로드
  geometry.ts  포트 회전·경로       power.ts     전력망 연쇄 계산
  flows.ts     흐름 전파·병목       persist.ts   저장·공유 URL
```

도메인 모델(요약): `FacilityType`(데이터 파일) / `ModuleInst`(배치 인스턴스, id·typeId·position·rot) /
`Connection`(fromModuleId·fromPort·toModuleId·toPort, 포트 키는 `in:<idx>`/`out:<idx>`).
전력은 별도 노드 타입 없이 `FacilityType.powerSource`(코어) + `powerRange`(중계기류)로 표현.

**중요:** 실제 게임 내 정확한 그리드 셀 크기, 설비 footprint, 정확한 분당 산출량은 공식 문서화되어 있지 않습니다.
`FacilityType` 데이터는 하드코딩하지 말고 `public/data/facilities.json`에서 관리하며, 사용자가 직접 보정할 수 있게 유지하세요.
표준 라인 = 컨베이어 벨트 최대 운송량 **2초당 1개 = 30/분** (파이프는 1초당 2개 = 120/분).

### 자원 명칭 (구 커뮤니티 용어 → 정식 명칭)

원소금→**오리지늄**, 철스러움→**페리움**, 구리움→**적동**. 주요 자원: 오리지늄 광물, 자수정 광석, 페리움 광석,
적동 광석, 청정수, 오염수, 산성 침적물, 정련 (광물), (광물) 가루, 고밀도 가루, 샌드리프 가루, 병, (광물) 부품,
협곡 배터리 / 무릉 배터리, 씨앗, 식물, 카본 조각, 식양, 중식양, 액화 식양, 양정, 양정 폐액, 식양 기체, 활성 대기.

## 4. 참고용 생산 체인 (레시피 시드 데이터로 활용)

- **협곡 배터리**: 채굴 → 정련로 → 부품 가공기(부품) + 분쇄기→연마기(고밀도 가루) → 포장기
- **치료 아이템**: 식물 분쇄(식물 가루) + 성형기(병) → 충진기
- **식물 무한 순환**: 재배기 2 + 씨앗 추출기 1 (협곡, 일명 재씨재) / 무릉은 1:1(재씨)
- **식양 (기체 공업 이전)**: 카본 다단계 가공 → 청정수와 반응 → 천화로
- **식양 (v1.4 기체 모드)**: 카본 조각 + 청정수 + 활성 대기 → 천화로에서 직접 합성 (공정 단축 —
  "기존 방식 vs 기체 모드" 비교 토글 만들면 재미있을 듯)
- **중식양**: 천화로에서 식양 기반 생산. 기체 정제 활용 시 양정 폐액 불필요
- **양정**: 확장 반응기(청정수+오염수+식양 → 양정 폐액 → +페리움 가루 → 양정)

## 5. 전력/연결 규칙 요약

- 프로토콜 코어(전력원) → 중계기(80m 범위 연쇄) → 전력 공급기(30m, 무선 공급) 순으로 전력 전달
- 설비는 활성 전력망 범위 안에 있어야 가동. 플래너에서는 반경 원(활성=노랑, 비활성=점선)으로 시각화,
  범위 밖 설비는 빨간 테두리 + ⚡ 경고 — 구현 완료
- 무릉 지역은 식양 계열 전력 설비만 사용 가능(일반 버전과 호환 불가) — 플래너에서는 구분만 표시
- v1.3부터 "예비 전원 시스템" 존재 → 완전 정전 방지 (시뮬레이션 불필요, 참고만)

## 7. 기술 스택 (확정)

- **Vite + TypeScript (Vanilla, 프레임워크 없음) + HTML5 Canvas**
- 데이터는 `public/data/facilities.json` 런타임 fetch — 패치 반영 시 코드 수정 없이 값만 갱신
- 상태 저장은 localStorage + 파일 export + URL 해시 공유. 백엔드 없음
- 배포: GitHub Pages (main push 시 Actions 자동 배포, `base: './'`)

## 8. 데이터 정확도에 대한 주의사항

- 설비/자원 **명칭**은 정식 한국어 서비스 기준(나무위키 대조, 2026-07-19)으로 검증됨
- **footprint·포트 개수/위치/종류(벨트·파이프)·전력 소비량은 실측값**: EnKAD(enkad.enka.network,
  enka.network의 Endfield 플래너)에 내장된 게임 클라이언트 추출 데이터에서 47종 변환 반영
  (예: 프로토콜 코어 9×9 입력14/출력6, 정련로 벨트3+파이프1). 변환 규칙: rotY 0/90/180/270 =
  흐름 방향 +z/+x/-z/-x, 입력은 흐름 반대편 변·출력은 흐름 방향 변, 화면 y=z
- **분당 산출량(rate)만 커뮤니티 추정치** (표준 라인 30/분 기준)
- 같은 리소스가 여러 포트에 있으면 병렬 레인 — flows.ts가 설비 단위로 합산 처리
- 기체 공업 4종(v1.4)은 추출 데이터에 아직 없어 추정치 유지
- 게임은 계속 패치되므로 하드코딩 금지, 데이터 파일 + 에디터 화면으로 사용자가 직접 갱신
